# Arush's Study Tracker

> A full-stack personal accountability system that converts YouTube comment likes into study hour obligations, syncs completed Todoist tasks to a GitHub-hosted session log via a Cloudflare Worker, and surfaces everything — live focus state, IST clock, 90-day heatmap, and real-time countdowns — in a single zero-dependency static page.

## The Problem

Most study trackers are self-reported. You log what you want, skip what you don't, and the only person keeping score is you.

This one isn't.

**Hours owed** are generated by YouTube comment likes — a public, verifiable number that no one controls but the people watching. I leave comments on long study-with-me videos, and each like on those comments adds a fixed number of hours to my study obligation (e.g. a like on a 4-hour video comment adds 4 hours). **Hours completed** are pulled automatically from Todoist task completions — no manual logging involved. The gap between the two is displayed as a large number at the top of the page.

The result is a live accountability contract between me and anyone who visits. If you liked a comment, you contributed to the debt. If the hours aren't climbing, the deficit is public.

## How It All Fits Together

There are three moving parts:

1. **`index.html`** — a static frontend with no framework and no build step. It fetches data from two sources and renders everything client-side.
2. **`data.json`** (hosted on GitHub) — the completed side of the tracker. Stores total hours done and the full session log. Written to automatically by the Worker when tasks are completed in Todoist.
3. **Cloudflare Worker** — a serverless edge function that sits between the browser and all external APIs. It holds every secret (API keys, tokens), handles caching, and exposes a clean set of `?action=` endpoints the frontend calls directly.

```
Browser  (index.html — no dependencies, no build step)
    │
    ├── GitHub Raw CDN  (data.json)
    │       └── session log, total hours completed
    │
    └── Cloudflare Worker  (edge-deployed, globally distributed)
            │
            ├── GET /                           → YouTube Data API v3
            │     Like counts across 18 tracked    (parallel fetch, KV-cached 5 min)
            │     comments → hours owed
            │
            ├── ?action=getFocusState           → KV read
            ├── ?action=setFocusState           → KV write  [admin only]
            │
            ├── ?action=getTodoistTasks         → Todoist REST API v1
            │     Active tasks in JEE project,     (cursor-paginated, KV-cached 3 min)
            │     sorted by tier + priority
            │
            ├── ?action=syncTodoistCompleted    → Todoist REST API v1
            │     Detects newly completed tasks,    → GitHub Contents API
            │     parses duration from description, → commits to data.json
            │     deduplicates via KV set
            │
            ├── ?action=getConfig               → KV read
            └── ?action=setConfig               → KV write  [admin only]
```

The browser holds no secrets. All credentials live exclusively in the Worker's environment variables.

## Features

### Hours Owed — YouTube Like Tracking

The Worker fetches like counts from 18 tracked YouTube comments using the YouTube Data API v3. Each comment has a `hoursPerLike` value set at the time it was added (matching the video length). The Worker multiplies likes × hoursPerLike across all comments and sums them into a total hours-owed figure.

To prevent inflated counts from stale YouTube API responses, stored like counts are **monotonically non-decreasing** — the Worker always takes `Math.max(apiCount, storedCount)`. If the API temporarily returns a lower number, the stored value holds.

Results are cached in Cloudflare KV for 5 minutes to stay well within the YouTube API's daily quota. The browser shows a live countdown to the next refresh.

### Hours Completed — Todoist Auto-Sync

When I complete a study task in Todoist, the Worker detects it on the next sync cycle (every 5 minutes). It:

1. Fetches recently completed tasks in the configured Todoist project
2. Parses the study duration from the task description or comments — supports formats like `2h`, `1.5h`, `90min`, `2h 30m`
3. Determines the correct date (see Date Priority below)
4. Appends the session to `data.json` via a GitHub Contents API commit
5. Marks the task ID as processed in KV to prevent double-counting on future syncs

A distributed lock (`todoist-sync-lock` in KV, 60-second TTL) prevents race conditions if multiple visitors trigger a sync at the same time.

### Date Priority

When deciding which date to log a session under:

1. **Task due date** — when the work was *scheduled* to happen (preferred)
2. **Completion timestamp** — when the task was actually marked done (fallback)

So if I complete Monday's assignment on Wednesday, it logs under Monday. The heatmap reflects when study *should have* happened, not just when I got around to finishing it.

### Focus State

A `setFocusState` endpoint (protected by an admin secret) lets me toggle an active focus session on or off by visiting a bookmarked URL from my phone. The state is stored in KV as `{ active, since, lastSeen }`.

The browser polls this every 60 seconds (30 seconds while a session is active) and renders:
- A **pulsing green dot** with elapsed time when focusing
- **"Last seen X ago"** when on a break

### Adaptive Sleep Notice

Between 12 AM and 8 AM IST, the clock card shows a contextual message:
- If a focus session is **active** → *"All-nighter mode — go get it!"* (green)
- If no focus session is running → *"Prolly sleeping — no updates 12 AM to 8 AM!"* (red)

Once the sleeping state is decided for a given night, it stays committed — it won't flicker back if I'm between focus sessions during a break. But if I start a new focus session, it overrides immediately.

### 90-Day Study Heatmap

A GitHub-style contribution grid covering the last 90 days. Each cell represents one day; colour intensity maps to hours studied that day across 6 levels (0 → 0–2h → 2–5h → 5–9h → 9–12h → 12h+). The grid is padded to full ISO weeks, only renders from the deploy date onwards, and each cell is hoverable with an exact hours tooltip. All date arithmetic runs in IST.

### Study Insights

- **Today's hours** — sessions logged for the current IST study day, plus day-of-year counter
- **This week's hours** — summed across the current ISO 8601 week, plus week number

"Today" is defined with a 2 AM cutoff — sessions logged before 2 AM are attributed to the previous calendar day, reflecting that late-night study belongs to the night it started.

### Active Todoist Tasks

The Worker fetches all active tasks in the configured project (including sub-projects), then sorts them into three tiers:

- **Tier 0** — due today
- **Tier 1** — overdue
- **Tier 2** — upcoming
- **Tier 3** — no due date

Within each tier, tasks are sorted by Todoist priority (P1→P4), then by due date. The top 8 are shown with colour-coded priority dots and overdue dates highlighted red.

### Like Sources Breakdown

A collapsible panel showing each tracked YouTube comment with its like count, total hours earned, and a `+Xh` badge that links directly to the comment. Sorted by contribution. Clicking the badge or title opens the comment for one-click liking.

## Engineering Decisions

| Problem | Decision |
|---|---|
| YouTube API daily quota is limited | Cache all 18 comment fetches in KV for 5 min; serve stale on quota error rather than failing |
| Concurrent sync requests could corrupt `data.json` | Distributed lock in KV with 60s auto-expiry TTL |
| Re-running sync could double-count sessions | Processed task IDs persisted in KV (capped at 1000 to prevent bloat); filtered before any write |
| Todoist returns paginated results | Full cursor-pagination loop on all endpoints — active tasks, completed tasks, task comments |
| YouTube occasionally returns stale (lower) counts | Monotonic storage: `Math.max(apiCount, storedCount)` — counts never decrease |
| Browser timezone varies by visitor | All date logic uses `toLocaleString("en-US", { timeZone: "Asia/Kolkata" })` — IST everywhere |
| GitHub CDN aggressively caches `data.json` | Cache-busted with `?t=Date.now()` on every fetch |
| Focus state shouldn't flicker during all-nighter breaks | `_nightDecided` flag committed once per night; only overridden when focus actively turns on |

## Cloudflare KV Namespace

| Key | Stored value |
|---|---|
| `cacheV2` | Full worker response payload (likes, hours, breakdown) with timestamp |
| `counts` | Per-comment like counts — monotonically increasing |
| `lastChange` | Most recent like event: label, URL, from/to counts, IST timestamp |
| `focus-state` | `{ active: bool, since: ISO, lastSeen: ISO }` |
| `todoist-cache` | Cached active task list with timestamp |
| `todoist-last-sync` | ISO timestamp of last completed-task sync window start |
| `todoist-processed` | Array of synced Todoist task IDs (deduplication set) |
| `todoist-sync-lock` | Write lock — present means sync in progress; 60s TTL |
| `config-target-project` | Todoist project name to filter against |

## Environment Variables (Worker)

| Variable | Used for |
|---|---|
| `YOUTUBE_API_KEY` | YouTube Data API v3 — comment like counts |

| Todoist completed sync | 5 min | `setInterval` |

## `data.json` Schema

```json
{
  "hoursCompleted": 86.5,
  "sessions": [
    {
      "topic": "Maths — Limits and Continuity",
      "hours": 2.5,
      "date": "28 Apr 2026",
      "loggedAt": "2026-04-28T14:00:00.000Z"
    }
  ]
}
```

`hoursCompleted` is recomputed as the sum of all session hours on every write. `loggedAt` is the Todoist completion timestamp (UTC). `date` is the human-readable IST date used for display.

## Local Development

No build step, no `npm install` for the frontend.

```bash
# Serve the static page
npx serve .
# or
python3 -m http.server 8080
```

To run the Worker locally with [Wrangler](https://developers.cloudflare.com/workers/wrangler/):

```bash
npm install -g wrangler
wrangler dev worker.js
```

Then update `WORKER_URL` in `index.html` to `http://localhost:8787`.

## Enabling Maintenance Mode

Uncomment this block near the top of `index.html`:

```html
<script>
  document.addEventListener("DOMContentLoaded", function () {
    document.getElementById("maintenance-screen").style.display = "flex";
    document.body.style.overflow = "hidden";
  });
</script>
```

This shows a full-screen animated overlay. Re-comment to restore the live page. No other changes needed.
